## What Went Well

- **Foundational conventions set deliberately.** `CatalogService` is now the spine: per‑method session lifecycle, flush‑before‑commit snapshot, `ValidationError` domain errors, the AD‑13 object error envelope, `_catalog_json_error` helper. Epics 2–10 inherit these.
- **Two‑tier testing discipline worked.** Unit tests on SQLite (`Base.metadata.create_all`) + Alembic migration round‑trip on a disposable MariaDB 11.8 container. Caught dialect differences (JSON/BLOB variants, Decimal, FK vs CHECK enforcement) before they shipped.
- **Migration playbook matured and held.** Hand‑write → chain from HEAD → verify on container → single `drop_table` on downgrade (PR #44 lesson) — reused cleanly across 1‑1, 1‑2, 1‑5.
- **Exploration agents → first‑pass‑green implementations.** 3 parallel agents before 1‑3; focused agents before 1‑4/1‑5. Mapping existing patterns (routes/service/templates, the `Photo` BLOB precedent, the `api_client` idiom) meant mirroring conventions, not reinventing.
- **Reviews earned their keep.** 1‑3's fresh‑context agents independently converged (3 ways) on the edit‑input‑loss bug; Copilot caught the MEDIUMBLOB off‑by‑one and the unproven‑deferral test on 1‑5.

## Recurring Themes (Challenges / Growth)

1. **Detached‑instance hazard — tracked and retired across three stories.** Surfaced as a test bug in 1‑2 (`expunge_all` → `DetachedInstanceError`, fixed to `expire_all`), flagged forward in 1‑3's `get_product` docstring, solved by design in 1‑4 (dedicated `get_purchases_for_product` query, never `product.purchases` navigation on a detached row).
2. **"Catch it as a `ValidationError`, never a raw DB error" — the single most recurring review finding.** 1‑3: 404‑vs‑500 (swallowed errors) and missing length validation. 1‑5: MEDIUMBLOB off‑by‑one (16,777,216 vs the 2²⁴−1 ceiling) and an unvalidated filename length. Root cause each time: an unguarded boundary in the service that detonated at the DB. **Not a knowledge gap — a checklist gap.**
3. **SQLite‑vs‑MariaDB enforcement split.** Learned concretely: SQLite ignores FKs (1‑2) but enforces CHECK constraints (verified before 1‑5). Reinforces the two‑tier testing discipline and per‑story notes on which invariants SQLite can/can't exercise.

## Action Items

1. **Bake a boundary‑validation checklist into the story‑creation guardrails.** For any service method that writes to the DB, enumerate and guard the boundaries as `ValidationError`s *before* the insert: string lengths (vs column widths), numeric ranges (vs column type ceilings — e.g. MEDIUMBLOB), and constraint preconditions (FK owner exists, XOR/uniqueness). Directly targets the #1 recurring review finding. — Owner: Amelia — Status: open
2. **Keep the two‑tier testing discipline explicit per story.** Continue unit‑on‑SQLite + migration‑round‑trip‑on‑MariaDB, and note in each schema story which invariants SQLite does/doesn't enforce (so DB‑level checks aren't assumed covered by the unit suite). — Owner: Amelia — Status: open
3. **Per‑request SQLAlchemy engine/pool creation — remains deferred.** Systemic (shared with the pre‑existing `InventoryService`); fix belongs at app scope covering both services. Epic 2 does not worsen it and it is not blocking, so it stays in `deferred-work.md` for a dedicated infrastructure change. — Owner: Amelia — Status: open (deferred)
